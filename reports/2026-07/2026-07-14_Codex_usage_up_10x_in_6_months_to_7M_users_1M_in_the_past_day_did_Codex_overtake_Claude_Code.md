# AI 트위터 요약
Agent RL 인프라: Prime Intellect의 Verifiers v1 및 장기 롤아웃
- Prime Intellect의 verifiers v1: Prime Intellect는 에이전틱 RL 및 평가를 위한 환경 스택을 대폭 재설계한 verifiers v1을 출시했습니다. 핵심 추상화는 환경을 태스크셋, 하네스, 런타임으로 분할하여, Johannes Hage가 강조하고 후속 심층 분석에서 다루었듯이, 이기종 실행 설정 전반에 걸쳐 코딩 및 컴퓨터 사용 에이전트를 위한 "자체 하네스 가져오기" 워크플로우를 명시적으로 지원합니다. 이 출시는 팀원들에 의해 수개월간의 인프라 현대화 작업으로 프레임되었으며, willccbb, mikasenghaas, xeophon의 풍부한 코멘트를 포함하여 주요 효율성 향상을 가져왔습니다.
- 기술적으로 중요한 이유: 가장 중요한 근본적인 변화 중 하나는 롤아웃 트레이스가 이제 메시지 DAG로 저장된다는 것입니다. 따라서 각 메시지는 전체 히스토리에 반복적으로 복사되는 대신 한 번만 저장됩니다. Prime Intellect에 따르면, 이는 턴 수에서 트레이스 성장을 O(n²)에서 O(n)으로 전환하여 장기 멀티모달 롤아웃 및 라우터 리플레이를 훨씬 더 실용적으로 만듭니다. 팀은 또한 구체적인 학습 구성을 주장했습니다: 사용자 제공 코딩 하네스에서 40턴 SWE 에이전트 태스크에 대해 1000 RL 스텝 동안 6개의 H200 노드를 사용하여 2일 이내에 100B 추론 모델을 학습시켰다는 것입니다 (willccbb). 이 주장은 vLLM의 생태계 지원으로 강화되었는데, vLLM은 verifiers의 롤아웃 경로가 vLLM에서 정확한 토큰 ID/로그 확률로 실행되어 서빙과 학습 간의 토큰화 드리프트를 방지한다고 언급했습니다.
코딩 에이전트, 하네스 설계 및 태스크당 비용 경쟁
- 하네스가 제품 표면이 되고 있습니다: 여러 게시물에서 모델 품질이 더 이상 유일한 차별화 요소가 아니라는 생각에 수렴했습니다. 하네스/오케스트레이터가 점점 더 결과를 결정하고 있습니다. threepointone의 강연은 "하네스가 앱이다"라고 요약되었으며, LangChain은 성공적인 에이전트 제품이 일반적인 래퍼가 아닌 태스크 전문 하네스에서 나올 것이라고 주장했습니다. Artificial Analysis 또한 장기 지식 작업에서 태스크당 비용이 점점 더 중요해지는 지표임을 강조하며, 토큰 가격만으로는 턴, 상세도, 캐시 적중률의 영향을 놓친다고 주장했습니다. 오픈 모델, 인퍼런스 시스템 및 양자화
- Transformers↔vLLM 통합으로 중복 모델 구현 작업이 제거됩니다: Clement Delangue는 주요 오픈 인퍼런스 유용성 개선 사항을 강조했습니다. Hugging Face Transformers 모델은 이제 vLLM에서 네이티브 속도로 실행될 수 있으며, 종종 수동으로 작성된 구현과 일치하거나 이를 능가합니다. 이것이 광범위하게 일반화된다면, 각 새로운 아키텍처를 두 번 구현해야 하는 오랜 부담(한 번은 연구/학습용, 한 번은 고성능 서빙용)을 줄이고 새로운 오픈 모델 아키텍처의 채택을 실질적으로 가속화할 수 있습니다.
- 양자화는 여전히 주요 레버리지입니다: waterloo_intern은 NVIDIA의 ModelOpt를 포함한 기존 접근 방식을 능가한다고 주장하는 새로운 양자화 방법을 미리 공개했습니다. 이 방법은 더 나은 레이어별 정밀도 할당을 더 빠르게 찾아내고, 더 공격적인 양자화와 더 높은 벤치마크 점수를 제공합니다. 이를 보완하여 Unsloth는 GGUF, NVFP4, FP8을 아우르는 LLM 양자화 및 배포에 대한 AWS 가이드를 발표했습니다. 또한 nrehiew_로부터 fp4 RL / fp4 서빙에 대한 실무자 의견이 있었는데, 이는 낮은 비트의 사후 학습이 제한된 품질 손실로 저렴한 서빙을 가능하게 할 수 있다고 주장했습니다.
- GLM-5.2 및 로컬/오픈 코딩 스택이 계속해서 인기를 얻고 있습니다: 여러 사용자들이 실제 워크플로우를 오픈 또는 세미-오픈 설정으로 옮겼다고 설명했습니다. juanjucm은 코딩 에이전트 워크플로우에 GLM-5.2를 사용하는 방법을 작성했으며, TheZachMueller는 실제 작업 파이프라인 하나를 Claude에서 GLM 5.2 NVFP4와 Kimi K2.7 Code NVFP4를 8xB200 노드에서 사용하는 스택으로 마이그레이션했다고 보고했습니다. 이를 통해 더 느린 실제 시간 레이턴시에도 불구하고 매우 저렴한 비용으로 더 밀도 높은 보고서를 얻을 수 있었습니다. nutlope 또한 GLM 5.2를 기반으로 재구축된 LlamaCoder v4를 출시했습니다.
에이전트 툴링의 보안, 프라이버시 및 데이터 제어
- Grok Build 코드 업로드 논란: 가장 중요한 보안 관련 소식은 IntCyberDigest와 hrkrshnn으로부터 나왔는데, 이들은 xAI의 Grok Build CLI가 코딩 태스크에 필요한 것보다 훨씬 많은 전체 리포지토리(개인 코드 및 비밀 포함)를 Google Cloud 버킷에 업로드하고 있다고 주장했습니다. 비판은 범위, 조용한 서버 측 완화, 그리고 불분명한 보존/삭제 보장에 집중되었습니다. 이는 에이전트 도구가 실제로 무엇을 전송하는지, 그리고 옵트아웃 UX가 와이어 레벨 동작과 왜 다를 수 있는지에 대한 더 광범위한 논의를 촉발했습니다.
- xAI의 응답은 ZDR 및 프라이버시 제어를 강조했습니다: SpaceXAI는 제로 데이터 보존을 사용하는 팀의 경우 트레이스 및 코드 데이터가 보존되지 않으며, API 키 사용은 ZDR을 준수하고, /privacy 명령을 통해 보존을 비활성화하고 이전에 동기화된 데이터를 삭제할 수 있다고 답변했습니다. 이는 일부 운영상의 질문에 답했지만, 기본 동작, 이전 업로드, 공개 규범에 대한 커뮤니티의 우려를 완전히 해소하지는 못했습니다.
- 신뢰 경계가 오픈 소스 대 클로즈드 소스 논쟁의 핵심이 되고 있습니다: 여러 게시물에서 이 사건을 넘어 대화가 확장되었습니다. mchiang0610과 jmorgan은 오픈 모델이 단순히 비용에 관한 것이 아니라 인간-AI 학습 루프에 대한 제어권과 기관 지식을 사내에 유지하는 것에 관한 것이라고 주장했습니다. Arav Srinivas는 ZDR 가용성이 Perplexity가 Grok 4.5를 Computer 하네스에 빠르게 통합한 이유 중 하나라고 말했습니다.
